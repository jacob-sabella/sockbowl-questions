## Resolver → permission mapping

### GraphQLController (`@QueryMapping`) — all → `packet:read`
- `getAllPackets`, `getPacketById`, `searchPacketsByName`

### PacketAuthoringController (`@MutationMapping`)
- `packet:create`: `createPacket`, `addTossupToPacket`, `addBonusToPacket`, `addBonusPart`
- `packet:update`: `renamePacket`, `setPacketDifficulty`, `updateTossup`, `reorderTossup`,
  `updateBonus`, `reorderBonus`, `updateBonusPart`, `reorderBonusPart`, `setTossupSubcategory`,
  `setBonusSubcategory`
- `packet:delete`: `deletePacket`, `removeTossupFromPacket`, `removeBonusFromPacket`, `removeBonusPart`
- `taxonomy:manage`: `createDifficulty`, `createCategory`, `createSubcategory`
- `question:generate`: `generateAndAddTossup` (AI-assist mutation that calls the generation
  service and adds the result to a packet; gated by the generation permission rather than
  `packet:create` since its distinguishing capability is invoking the AI generator)

Beyond the resolvers explicitly named in the task spec, this controller also has
`reorderTossup`, `reorderBonus`, `reorderBonusPart` (reordering → mapped to `packet:update`,
following the "set*/rename*→update" pattern) and `setTossupSubcategory`, `setBonusSubcategory`
(also `packet:update`).

### PacketGenerationController (REST, `/api/packets/generate`)
- `question:generate`: `generatePacket` (the only endpoint in this controller)

## Test approach

The full Spring context requires Neo4j, unavailable in the test environment, so a full
`@SpringBootTest` was avoided. Instead: a `@WebMvcTest(controllers = SecurityProbeController.class)`
slice, importing only `SecurityConfig`, with a throwaway top-level `@RestController`
(`SecurityProbeController`) exposing two `@PreAuthorize`-guarded probe endpoints.

**Gotcha found and fixed:** the task's suggested test had `Probe` as a *nested static class inside
the test class itself*. That silently fails in this Spring Boot 4.1 setup — Spring's
`TestTypeExcludeFilter` excludes classes whose enclosing class is a JUnit test class from the
slice's component scan, so the nested controller was never registered as a bean and every request
404'd (looked like a routing bug, not an auth bug) instead of exercising `@PreAuthorize`. Moving the
controller to a top-level class (`SecurityProbeController.java`) fixed it. Also had to add
`spring-boot-webmvc-test` as a test dependency — in Spring Boot 4.1 `@WebMvcTest` moved out of
`spring-boot-test-autoconfigure` into its own module (package also moved from
`org.springframework.boot.test.autoconfigure.web.servlet` to
`org.springframework.boot.webmvc.test.autoconfigure`).

Three tests, all green:
- `unauthenticated_is_401` — no auth → 401
- `wrong_authority_is_403` — JWT with `packet:create` hitting a `packet:read`-gated endpoint → 403
- `right_authority_is_200` — JWT with `packet:read` → 200

## Concerns / notes for follow-up

- No integration test exercises the guest-mode (`NoSecurityConfig`) path or a real Keycloak token
  end-to-end — only the `SecurityConfig` (auth-on) wiring is unit-tested via the MockMvc slice.
  Manual/integration verification against an actual Keycloak instance is still recommended before
  enabling `SOCKBOWL_AUTH_ENABLED` in any real environment.
- `generateAndAddTossup`'s mapping to `question:generate` (rather than `packet:create`) is a
  judgment call beyond the explicit spec — flagging in case the intended permission model wants
  AI-assisted mutations to also require `packet:create`.
